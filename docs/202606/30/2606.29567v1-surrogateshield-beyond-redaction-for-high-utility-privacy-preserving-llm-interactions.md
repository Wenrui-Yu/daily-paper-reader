## 摘要
基于LLM的助手将用户查询逐字传输到用户无法审计或控制的第三方API端点。当这些查询包含个人身份信息（PII）时，数据将持久存储在远程基础设施上，面临泄露、传票或政策变更的风险。占位符脱敏（当前主流的缓解措施）以牺牲语义连贯性为代价抑制PII，产生结构退化的查询以及相应退化的响应。我们提出SurrogateShield，一个客户端代理，在传输前将检测到的PII替换为本地生成的类型一致的替代值，并在响应中恢复原始值。没有真实的PII跨越网络边界。检测通过三级级联（PatternScan、EntityTrace和ContextGuard）运行，涵盖基于Sweeney的k-匿名框架的22种PII类型和准标识符组合。替代到原始值的映射被密封在AES-256-GCM加密的每个对话ShadowMap中，该映射从不离开设备。在1,124个查询的语料库上的评估表明，该级联可靠地检测到PII，实现了98.87%的整体F1分数。替代替换在语义实用性方面显著优于占位符脱敏，使BERTScore（roberta-large）从81.59%提高到94.85%，提升了13.26个百分点。在该语料库中，本地流水线在所有测试的查询类型上限制了真实PII的传输；在100个查询的对抗性试验中，一个经过提示的LLM对手未能从替代替换的消息中恢复出任何原始值。

## Abstract
LLM-based assistants transmit user queries verbatim to third-party API endpoints that lie outside the user's audit or control. When those queries contain personally identifiable information (PII), the data persists on remote infrastructure subject to breach, subpoena, or policy change. Placeholder redaction (the prevailing mitigation) suppresses PII at the cost of semantic coherence, producing structurally degraded queries and correspondingly degraded responses.   We present SurrogateShield, a client-side proxy that substitutes detected PII with locally generated, type-consistent surrogate values prior to transmission and restores originals in the response. No real PII crosses the network boundary. Detection runs through a three-stage cascade (PatternScan, EntityTrace, and ContextGuard) covering 22 PII types and quasi-identifier combinations grounded in Sweeney's k-anonymity framework. Surrogate-to-original mappings are sealed in an AES-256-GCM encrypted per-conversation ShadowMap that never leaves the device.   Evaluations on a 1,124-query corpus demonstrate that the cascade reliably detects PII, achieving an overall F1 score of 98.87%. Surrogate substitution substantially outperforms placeholder redaction in semantic utility, yielding a 13.26 pp improvement in BERTScore (roberta-large), from 81.59% to 94.85%. Within this corpus, the local pipeline restricted real PII transmission across all tested query types; in a 100-query adversarial trial, a prompted LLM adversary recovered no original values from surrogate-substituted messages.

## 1. 核心问题与整体含义（研究动机和背景）
- **问题**：LLM 助手将用户查询逐字发送至第三方 API 端点（如 OpenAI），这些查询常含个人身份信息（PII），数据持久存储在远程服务器上，面临泄露、传票、内部威胁或政策变更的风险。用户无法审计或控制对方操作。
- **现有局限**：当前主流缓解措施是**占位符脱敏**（如用 `[PERSON]` 替换姓名），虽能防止真实 PII 传输，但严重破坏语义连贯性，导致 LLM 响应质量下降（例如回复以 `[PERSON]` 开头）。
- **整体含义**：本文认为隐私要求是“真实 PII 值从不离开设备”，而非“PII 位置的槽必须为空”。因此提出**类型一致的替代值替换**方案，在保证隐私的同时保留语义结构，实现高实用性的隐私保护 LLM 交互。

## 2. 方法论核心思想、关键技术细节
### 核心思想
- **SurrogateShield** 是客户端代理（client-side proxy），拦截用户查询，在传输前用本地生成的类型一致替代值替换 PII，并将映射 AES-256-GCM 加密存储于设备，响应中恢复原始值。确保真实 PII 永不跨网络边界。
### 关键技术细节
- **三级检测级联（SentinelLayer）**：
  1. **PatternScan**：基于正则与校验和（如 Luhn 校验、ABA 校验和）检测 16 种结构化 PII（SSN、信用卡、电话等）。
  2. **EntityTrace**：使用 spaCy `en_core_web_lg` NER，置信度≥0.85 的实体直接确认，0.60-0.85 的转交 ContextGuard。
  3. **ContextGuard**：本地执行 DistilBERT NER（dslim/distilbert-NER），验证边界实体并独立检测遗漏实体。
- **后处理四步**：结构 ORG 正则修复、Email-用户名重新分类、Person 去重、主题地理实体过滤（区分个人位置与查询话题）。
- **准标识符风险检测**：基于 Sweeney 的 k-匿名框架，监控 10 种组合（如 ZIP+DOB+Gender），满足条件时发出警告并替换所有字段。
- **服务查询智能**：通过 15 种模式区分服务查询（如找餐厅、导航）与个人位置信息，对地址仅进行微偏移（±2-8 号），敏感性主题（医疗、移民等）强制完全匿名化。
- **替代值生成（MimicGen）**：使用 Faker 生成类型一致值，每会话唯一（通过 used_surrogates 集合），50 次碰撞后加随机后缀。
- **加密映射存储（ShadowMap）**：每对话独立 AES-256-GCM 加密文件，密钥由 HKDF-SHA256 从设备级密钥派生。双对话历史：display history（显示给用户，含真实值）与 api_history（仅替代值，发送给 LLM）。
- **响应恢复（ResolvePass）**：三遍恢复：精确匹配、组件匹配（仅对未解析替代值）、模糊匹配（rapidfuzz partial_ratio），记录 exact/fuzzy/miss 分类。
- **隐私感知 RAG（可选）**：本地 ChromaDB + all-MiniLM-L6-v2，文档经 SentinelLayer 后索引，查询匿名化后检索。

- **完整性**：覆盖 22 种 PII + 准标识符组合；三级级联设计充分权衡规则与 NER；服务查询智能处理边界情况。

## 8. 不足与局限
- **检测覆盖有限**：仅 22 种 PII 类型，无法枚举所有格式（如医疗记录号、特定辖区 ID）。隐含 PII（写作风格、罕见疾病等）不在检测范围。
- **数据集偏差**：使用合成 PII 和人工构造查询，虽模拟口语化，但真实用户查询的语法多样性和格式不规范可能降低检测召回。作者承认需伦理许可才能获取真实数据集。
- **替代值分布局限**：Faker 偏重西方名字和 US 格式，非英语姓名可能风格不一致，可能暴露匿名化。
- **服务查询边界启发式**：15 个模式可能误分类，导致欠匿名化或过匿名化。
- **信任模型边界**：ShadowMap 加密仅防文件级访问，不防内核级恶意软件或内存读取。
- **未处理威胁**：未应对 LLM 隐写泄露、流量分析、模型反演等更广泛攻击。
