## 摘要
全同态加密（FHE）允许直接在加密数据上执行计算而无需解密，为敏感数据分析提供了强大的隐私保障。这一能力对于安全云计算、金融和医疗等隐私敏感型应用至关重要。然而，FHE方案的复杂性阻碍了其实际应用。为了使FHE为广大开发者所用，新一代专用框架应运而生，将高级FHE程序转化为复杂的FHE操作，引入了新的编程范式。然而，FHE框架固有的复杂性使其容易出现错误的实现逻辑。与纯粹的程序崩溃不同，这些框架中的逻辑错误可能悄然破坏加密计算，导致FHE增强型应用中出现严重的财务损失和安全漏洞。在本工作中，我们提出了HERTA，这是首个专为FHE框架设计的自动化测试工具。HERTA利用蜕变测试来揭示多层FHE软件栈中深层的实现错误和漏洞。为此，我们设计了一组新颖的蜕变关系（MR），这些关系专门从FHE语义中推导而来。这些MR针对流水线中最具挑战性的方面，使得无需人工真值即可进行自动化正确性测试。我们对HERTA在三个领先行业框架上的评估发现了21个先前未知的错误，其中多个已被开发者确认并修复。此外，我们的危害分析揭示了这些错误对基于FHE的服务的完整性和可用性所构成的严重安全影响。

## Abstract
Fully homomorphic encryption (FHE) allows computations to be performed directly on encrypted data without decryption, offering strong privacy guarantees for sensitive data analysis. This capability is important for privacy-sensitive applications like secure cloud computing, finance, and healthcare. The complexity of FHE schemes, however, has hindered their practical adoption. To make FHE accessible to a broader range of developers, a new generation of specialized frameworks has emerged to translate high-level FHE programs into complex FHE operations, introducing a new programming paradigm. However, the inherent complexity of FHE frameworks makes them prone to incorrect implementation logic. Unlike mere crashes, logic bugs in these frameworks can silently corrupt encrypted computation, potentially leading to severe financial losses and security vulnerabilities in FHE-enhanced applications.   In this work, we introduce HERTA, the first automated testing tool tailored for FHE frameworks. HERTA leverages metamorphic testing to uncover deep-seated implementation bugs and vulnerabilities across the multi-layered FHE software stack. To that end, we design a set of novel metamorphic relations (MRs) derived specifically from FHE semantics. These MRs stress the most challenging aspects of the pipeline, enabling automated correctness testing without the need for a manual ground truth. Our evaluation of HERTA on 3 leading industry frameworks discovered 21 previously unknown bugs, several of which have already been confirmed and fixed by developers. Furthermore, our hazard analysis reveals the critical security impact these bugs pose to the integrity and availability of FHE-based services.

- **研究动机**：全同态加密（FHE）框架通过高层程序抽象降低了使用门槛，但其多层软件栈（前端解析、中间优化、后端加密执行）极其复杂，容易出现实现逻辑错误。这些错误往往表现为“静默错误”（silent errors）——不会导致崩溃或挂起，但悄然破坏加密计算结果，在金融、医疗等隐私敏感场景中可能引发严重的安全漏洞（如财务损失、误诊）。
- **核心问题**：现有软件测试方法（如传统编译器测试、通用模糊测试）缺乏对FHE领域知识的感知，无法系统性地探测FHE框架特有的噪声管理、参数合成、跨方案一致性等复杂问题。目前尚无针对FHE框架的自动化测试工具。
- **贡献**：本文提出 **HERTA**，首个专为FHE框架设计的自动化测试工具，利用基于FHE语义的蜕变测试（Metamorphic Testing）发现深层逻辑错误和崩溃。

## 2. 方法论：核心思想与关键技术细节

### 核心思想
- **蜕变测试（MT）**：通过定义输入变换后的输出必须满足的不变性（蜕变关系，MR）来验证程序正确性，避免对人工真值的依赖。
- **FHE特定MR设计**：针对FHE框架的三个关键阶段设计了三类蜕变关系，分别处理程序前端变换、编译配置突变、后端/方案重定向。

### 关键技术细节
#### (1) FHE感知的种子生成
- 定义上下文无关文法 Λ_HERTA，抽象FHE程序语法，区分隐私类型（plaintext/encrypted）和间接操作（π，如max、sin等缺乏直接FHE原语的操作）。
- 生成算法动态调整概率权重：早期偏重算术运算（如乘法）以积累乘法深度，后期转向非直接操作以增加功能多样性，从而挑战框架的噪声管理和参数合成。

#### (2) MR1：类型与数据流突变
- **类型突变**：改变变量位宽、标量与张量布局切换、明文/密文互换。
- **数据流突变**：代数表达式重构（如m×m→m²）、冗余操作插入（加0乘1）、等价非线性替换（如v→max(v,MIN_VAL)）。
- 目标：检测数据表示、打包、噪声管理中的静默错误。

#### (3) MR2：编译配置突变
- 对适配策略（C_ada）、优化设置（C_opt）、加密参数（C_crypto）进行突变，保持程序P不变，验证输出一致性。
- **挑战**：配置空间巨大（数十个参数，组合暴增）。采用**优先组合测试**：
  - 先通过2-way pairwise覆盖生成候选套件（平衡检测效率与覆盖率）。
  - 再通过**使用感知剪枝**：仅选择与程序中出现的操作符相关的参数。
  - 最后通过**多样性引导选择**：每次选择与已执行配置汉明距离最大的配置，最大化探索范围。

#### (4) MR3：后端与方案突变
- 保持程序不变，切换后端（CPU/GPU、OpenFHE/SEAL等）或加密方案（BFV/BGV/CKKS等），验证输出一致性。
- **自适应容错检查**：对于精确整数方案（如BGV、BFV）强制严格相等；对于近似算术方案（如CKKS）采用1%相对误差容忍度（基于1000个已知正确测试的统计分析）。

#### 整体流程
1. 生成种子程序P及其默认配置Γ。
2. 对P应用MR1得到P′，验证F(P,Γ)=F(P′,Γ)。
3. 对配置Γ应用MR2得到Γ′，验证F(P,Γ)=F(P,Γ′)。
4. 对方案/后端应用MR3得到Γ″，验证F(P,Γ)≈F(P,Γ″)（根据方案类型决定容忍度）。
5. 报告崩溃或输出不一致为漏洞。

## 7. 优点（亮点）

- **领域首创**：首次针对FHE框架提出系统性自动化测试方法，填补空白。

## 8. 不足与局限

- **语言特性覆盖不全**：仅针对FHE框架的数学API层，未覆盖高层领域特定接口（如HELayers的ONNX输入），虽然影响核心根基，但完整性有欠缺。
- **种子生成缺乏复杂静态分析**：采用轻量级迭代重试替代深度数据流/噪声分析，虽有效性高（98.93%），但可能遗漏某些需要精确静态预测的边界情况。
- **MR2发现漏洞较少**：作者承认当前FHE框架的配置空间尚在发展，MR2的价值可能随框架成熟而提升；目前仅发现1个独特漏洞，贡献度相对有限。
- **与现有方法的对比有限**：仅与Domato对比，缺少与其他编译器测试方法（如差分测试、符号执行）在FHE场景下的比较，但后者因领域特异性可能难以直接应用。

- **评估框架的版本局限**：测试基于特定版本，FHE框架快速迭代中可能已有修复或引入新bug，结论时效性有限。
