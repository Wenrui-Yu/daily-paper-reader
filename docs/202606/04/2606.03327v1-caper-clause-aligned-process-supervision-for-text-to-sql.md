## 摘要
Text-to-SQL系统通常通过查询级别的执行正确性来评估，但这种终端信号对于哪个中间SQL决策导致了成功或失败提供的指导甚少。令牌级密集监督同样不适用：SQL令牌与完整的语义决策不对齐，可能惩罚执行等价查询，并且难以大规模可靠标注。因此，我们提出CAPER，通过在SQL抽象语法树上进行反事实干预自动推导出子句级监督，从而为奖励建模实现根因错误定位；所得数据用于训练轻量级子句过程奖励模型CAPER-9B，该模型为策略优化和候选验证提供子句边界反馈。在BIRD和Spider数据集上的实验表明，子句对齐的监督不仅提高了执行准确率，相比GPT-5.4相对EX提升高达15.3%，而且增强了故障定位能力，在保留的失败案例上达到了84.53%的准确率和90.60%的MRR。我们的项目页面位于https://github.com/banrichard/RL-NL2SQL。

## Abstract
Text-to-SQL systems are typically evaluated by query-level execution correctness, but this terminal signal provides little guidance about which intermediate SQL decision caused success or failure. Token-level dense supervision is also ill-suited: SQL tokens do not align with complete semantic decisions, can penalize execution-equivalent queries, and are difficult to label reliably at scale. We therefore propose CAPER, which automatically derives clause-level supervision via counterfactual intervention on the SQL abstract syntax tree, enabling root-cause error localization for reward modeling; the resulting data is used to train CAPER-9B, a lightweight Clause-PRM that provides clause-boundary feedback for policy optimization and candidate verification. Experiments on BIRD and Spider show that clause-aligned supervision not only improves execution accuracy, achieving up to a 15.3% relative EX improvement over GPT-5.4, but also strengthens failure-localization capability, reaching 84.53% accuracy and 90.60% MRR on held-out failures. Our project page is at https://github.com/banrichard/RL-NL2SQL.

# 论文深度分析报告：CAPER

## 1. 论文的核心问题与研究背景

本论文聚焦于 **Text-to-SQL 任务中的过程监督粒度问题**，核心矛盾在于：

- **查询级执行反馈过于粗糙**：现有系统仅依据最终 SQL 执行结果是否正确来评估。这种“终端信号”无法定位是 WHERE 子句、JOIN 路径还是聚合函数等哪个中间语义决策导致了失败，造成“信用分配难题”。
- **Token 级监督存在语义不对齐**：直接对 SQL 令牌进行逐词奖励存在根本缺陷。单个 Token（如逗号、别名）不构成完整语义决策；语法等价但写法不同的查询可能被错误惩罚；且难以获得大规模、可靠的 Token 级标注。

因此，论文主张过程监督的**正确粒度应介于查询级与令牌级之间，即“子句/子句级”**。子句作为语义完整的 SQL 结构单元，能自然承载中间决策的成败信息，同时避免了 Token 级的碎片化问题。

## 2. 方法论：CAPER 框架

CAPER 方法论包含三部分：自动子句级奖励标注、子句级过程奖励模型训练、以及基于该模型的策略优化。

**核心思想：** 利用 **反事实干预** 在 SQL 抽象语法树上自动定位导致执行失败的“根因节点”，并将该节点错误通过拓扑感知的方式传播、聚合为子句级奖励信号。

**关键技术细节：**

- **子句单元定义**：将 SQL 语句解析为顶层子句序列，包括 `SELECT`, `FROM-JOIN`, `WHERE`, `GROUP-HAVING`, `ORDER-LIMIT` 等。
- **自动标注流程 (Automatic Clause-Level Reward Annotation)**：
  1.  **轨迹收集与分类**：使用监督微调策略模型生成 SQL 轨迹，根据执行结果正确与否划分为成功集与失败集。
  2.  **成功轨迹的故障注入 (Fault Injection)**：对成功轨迹的 AST 进行类型保持的变异（如列替换、聚合函数替换等），生成执行错误的新轨迹，从而制造出已知根因的失败案例。
  3.  **失败轨迹的反事实干预 (Counterfactual Intervention)**：对自然产生的失败轨迹，通过对比预测 SQL 与金标准 SQL，定位最早产生分歧的子句。然后执行 AST 级的“最小编辑脚本”来精确定位根因节点。
  4.  **拓扑感知的节点级奖励 (Topology-Aware Node-Level Reward)**：对 AST 每个节点赋予惩罚。根因节点本身惩罚最大，其依赖路径上的上游祖先节点惩罚按高斯距离衰减，无关分支则被掩码为零奖励。公式为：
      $$ R_{\text{node}}(v_i; v^*) = -\alpha M(v_i, v^*) \exp\left(-\frac{d^2(v_i, v^*)}{2\sigma_d^2}\right) $$
      （其中 $M(v_i, v^*)$ 为因果掩码，$d$ 为最短路径距离）
  5.  **子句级聚合 (Clause-Level Aggregation)**：通过取最小值将节点惩罚聚合到子句单元，使得包含根因节点的子句获得强负信号。
- **子句级过程奖励模型 (Clause-PRM)**：基于上述标注的偏好数据，训练一个参数为 $\psi$ 的奖励模型 $f_\psi(x, p_k, u_k)$，评估在当前上下文字句 $p_k$ 下生成下一子句 $u_k$ 的正确性。训练目标为带边际加权的偏好排序损失函数。
- **策略优化 (Policy Optimization)**：将 SQL 生成视为 MDP，在每个子句边界处由 Clause-PRM 提供过程奖励，并在最终子句处叠加执行正确性奖励。使用该复合奖励，通过 **GRPO** 算法优化策略模型。

## 6. 主要结论与发现

1.  **子句级过程监督显著提升端到端性能**：在 Qwen3.5-9B 基座上，GRPO-Clause 在 BIRD 和 Spider 上都远超 SFT 基线和其他 GRPO 变体，其 Majority Vote@8 的准确率提升尤为显著。
2.  **令牌级密集奖励并非有效解决方案**：GRPO-Token 的性能不仅没有普遍超越 GRPO-Sparse，甚至在多数情况下表现更差，证明了简单扩散奖励信号并不会带来增益。
3.  **强大的故障定位能力**：CAPER-9B 能够以极高精度定位失败 SQL 的根因子句，显著优于自调试和仅执行奖励模型等基线，表明了其学习到了结构化的错误归因知识。
4.  **作为插件式验证器有效**：Clause-PRM 可以作为一个与生成器解耦的候选验证器，为闭源黑盒模型生成的多个 SQL 候选进行重排序，稳定提升多数投票的性能。

## 7. 优点与亮点

- **创新的监督粒度设计**：首次系统性地论证并实践了 SQL 过程监督中“子句”这一粒度的优越性，填补了查询级和 Token 级之间的巨大鸿沟。
- **全自动的弱监督数据构建**：提出的反事实干预和故障注入流程完全自动化，无需昂贵的人工标注，即可从查询级执行信号中提取出精准的子句级错误归因，具有极佳的扩展性。
- **根因定位的可解释性**：通过 AST 拓扑感知的奖励传播机制，模型不仅能知道“错了”，还能指出“哪错了”，增强了系统的可解释性和可调试性。
- **实验论证扎实**：通过详细的消融研究和对比实验，清晰地展示了各组件的作用，并且实验设置严谨，结论可信。

## 8. 不足与局限

- **只限于顶层子句粒度**：CAPER 当前仅将 SQL 分解到顶层子句。对于嵌套在子查询内部的错误，或者一个 WHERE 条件中多个谓词之一的错误，该方法会将其归入整个父级子句，无法进行更精细的定位。
- **标注流程的计算开销**：自动标注流程涉及基于模型的多次采样、AST 解析、反事实修复/注入以及重复的数据库执行，计算开销高于直接使用稀疏终端奖励。
- **领域泛化性未验证**：所训练的奖励模型和策略模型强依赖于训练数据集的 SQL 语境和数据库模式，其在新领域或不同 SQL 方言上的泛化能力尚待考查。
- **限于 SQL 生成而非推理链**：方法聚焦于最终的 SQL 生成任务，并未涉及中间推理步骤的监督，其思想能否直接推广到其他结构化文本的生成还有待研究。
