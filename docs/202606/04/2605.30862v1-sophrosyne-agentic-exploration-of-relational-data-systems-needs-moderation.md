## 摘要
由大语言模型驱动的 Text2SQL 代理通过工具调用探索数据系统，在生成查询之前将自然语言意图转化为 SQL。然而，为了确保访问的安全性和范围可控，数据系统构建了具有明确 API 界面的环境。我们研究并分类了当前暴露的这些 API，将其分为粗粒度和细粒度两类，并认为在它们之间做选择呈现了在成本高效的探索和准确的 SQL 生成之间的根本权衡。大多数数据系统暴露细粒度 API，但这无意中使代理处于不利地位：它们过度探索，将无关的模式元素纳入查询构建中，从而产生不准确的结果。我们认为，遏制过度探索是有效利用这些 API 界面的关键，并提出了 Sophrosyne，一个通过指令增强 API 响应以引导代理探索过程的数据系统环境。初步结果显示，指令将过度探索减少了 4.6 倍，并将准确性提高了最多 12.4%（约 4 个百分点）。

## Abstract
Text2SQL agents powered by LLMs translate natural language intent into SQL by exploring the data system through tool calls before formulating the query. However, to ensure secure and scoped access, data systems construct environments with explicit API surfaces. We study and categorize these APIs exposed today as either coarse-grained or fine-grained and posit that choosing between them presents a fundamental tradeoff between cost-efficient exploration and accurate SQL generation. Most data systems expose fine-grained APIs, but this inadvertently disadvantages agents: they over-explore, incorporating irrelevant schema elements into their query formulation and produce inaccurate results. We argue that curbing over-exploration is key to the effective use of these API surfaces, and propose Sophrosyne, a data system environment that augments API responses with directives that guide the agent's exploration process. Initial results show that directives reduce over-exploration by 4.6x and boost accuracy by up to 12.4% (approx. 4 percentage points).

## 一、论文的核心问题与整体含义

- **研究动机**：由大语言模型（LLM）驱动的 Text2SQL 智能体通过调用数据系统 API 探索表结构，再将自然语言问题转换为 SQL 查询。为保证安全与权限控制，现代数据系统普遍采用细粒度 API 暴露表名与结构，供智能体逐步探查。
- **核心问题**：细粒度 API 虽然能按需获取模式信息，但也导致智能体**过度探索（over-exploration）**——在构建查询时纳入大量无关的表或列，既浪费 token 资源，又降低 SQL 生成准确性。
- **整体含义**：数据系统 API 界面在探索效率与查询准确性之间存在根本权衡，必须引入**节制机制**以抑制过度探索，从而在成本与效果上取得平衡。该论文以古希腊概念“Sophrosyne”（节制）命名其解决方案，强调智能体需要“适度”探索。

## 二、论文提出的方法论

- **核心思想**：在细粒度 API 的响应中加入**指令（directives）**，告知智能体哪些表可能是无关的，引导其避开这些表，从而减少不必要的探索步骤。
- **指令生成流程**：
  1. **离线阶段**：构建一个混合检索索引 `HI_col`，综合语义相似度（IVF 索引）与关键词 BM25 得分，对各个列名建立索引。嵌入模型使用 `text-embedding-3-large`。
  2. **在线交互**：当智能体调用 `list_tables` 时，Sophrosyne 根据用户问题及领域知识库，先利用一个低成本模型（GPT-5.4-nano）从自然语言描述中**推测可能出现的列名**。
  3. 用推测出的列名在 `HI_col` 上检索，得到一组与问题相关的表。
  4. 将所有表名中**未出现在相关表集合中的**标记为无关表，作为指令随 `list_tables` 响应返回（如“Irrelevant tables: feedback, client”）。
- **系统实现**：Sophrosyne 作为一个 MCP 服务器，暴露与主流数据系统类似的细粒度 API（list_tables, describe_table, get_join_info, read_query, submit_query），仅在 `list_tables` 的响应中加入指令字段。

## 六、论文的主要结论与发现

- **过度探索是细粒度 API 的固有缺陷**：代理在细粒度 API 下会进行最多 65% 的额外探索，且模型推理强度提升对此无明显缓解。
- **指令有效抑制过度探索**：Sophrosyne 在不同模型上稳定降低过度探索，最多减少 4.6 倍；指令越准确，抑制效果越好。
- **过度探索直接影响准确性**：含无关表的失败查询数随过度探索减少而下降，Sophrosyne-Oracle 几乎消除这类错误，验证了因果关系。
- **成本优化**：通过减少不必要的模式探索，Sophrosyne 降低了输入 token 消耗（Sonnet 模型成本节省明显），仅在个别情况下因模型多余行为增加少量输出 token。

## 七、优点

- **问题新颖**：首次系统研究数据系统 API 粒度对 Text2SQL 智能体行为的影响，并将“过度探索”识别为核心权衡。
- **方法普适与轻量**：Sophrosyne 以非侵入式的方式在现有细粒度 API 中插入指令，无需修改智能体内部逻辑，可与任何支持 MCP 的代理或已有工作流集成。
- **实验设计严密**：通过 Oracle 设置确立理论上界，分离了指令质量的影响；多模型、多维度的评估使结论稳健。
- **经济可行**：离线索引成本极低，在线推理增加的开销仅 8 美分，且可通过减少 token 消耗回本甚至获益，具备实际部署价值。

## 八、不足与局限

- **指令准确率仍有提升空间**：当前机制平均召回率 82.5%，未能完全消除过度探索，部分场景仍需更精准的指令计算。
- **对模型指令遵循能力敏感**：即使提供完美指令，部分模型仍会使用无关表（如通过 `get_join_info` 获取关联信息后依然引入），说明指令作用的发挥依赖于模型的跟随能力。
- **长期上下文中指令影响力衰减**：指令在对话早期给出，若智能体上下文变长，其引导效应可能下降，未能探讨需重复提醒的机制。
- **实验场景局限于单数据源**：研究集中在单一数据库，多数据源或联邦查询场景下的过度探索问题更严峻，但尚未涉及。
- **性能提示方向未展开**：文中虽提到指令可提供性能优化建议（如索引使用），但现有版本仅聚焦探索控制，未验证其对查询执行效率的额外收益。
