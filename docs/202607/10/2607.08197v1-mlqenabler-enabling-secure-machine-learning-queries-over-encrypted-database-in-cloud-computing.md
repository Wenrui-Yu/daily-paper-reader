## 摘要
在云计算中，公共云服务提供商（CSP）可以提供云存储作为主要服务，同时通过使用客户存储在云端的数据，提供基于机器学习的附加服务。这种商业模式扩展了云计算的边界，并带来了新的业务增长可能性。尽管前景广阔，但由于公共商业云并非完全可信，该模型也带来了安全隐患。例如，公共商业云可能将客户的敏感数据出售给政府或其他公司。为了解决安全问题，一个直接的方法是要求客户在将数据集外包到云端之前对数据进行加密。然而，如果数据库被正式加密，那么数据库中只包含伪随机数，使得无法在其上启用机器学习。在本项目中，我们提出了MLQENABLER（机器学习查询启用器）方案，以实现对云存储中加密数据库的安全机器学习查询。MLQENABLER采用索引辅助方法，同时实现安全性和机器学习能力。我们的初步实验表明，MLQENABLER在仅轻微降低机器学习性能的同时，达到了可接受的安全水平。

## Abstract
In cloud computing, the public cloud service providers (CSPs) can provide cloud storage as the primary service while providing additional machine learning (ML)-based services by using the clients' data in storage. This business model extends the border of cloud computing services and brings in new business growth possibilities. Although it is promising, the model also brings in security concerns since the public commercial cloud cannot be fully trusted. For example, the public commercial clouds may sell clients' sensitive data to the government or other companies. To address the security concerns, an immediate solution is to require clients to encrypt their datasets before outsourcing to the cloud. However, if a database is formally encrypted, then the database contains only pseudorandom numbers, making it impossible to enable ML over it. In this project, we propose MLQENABLER (ML Queries Enabler) scheme to enable secure ML queries over encrypted database in cloud storage. MLQENABLER employs an index-aid approach to achieve security and ML capability simultaneously. Our initial experiments show that MLQENABLER achieves an acceptable security level while incurring only a slight ML performance degradation.

## 1. 核心问题与整体含义（研究动机和背景）
- **核心问题**：在云计算中，公共云服务提供商（CSP）提供云存储服务的同时，希望利用客户存储的数据提供额外机器学习（ML）服务（如推荐系统）。然而，公共云并非完全可信，存在泄露客户敏感数据的风险（如出售数据、内部员工窃取、黑客攻击等）。
- **现有解决方案的局限性**：
  - **全同态加密（FHE）**：效率极低，训练深层神经网络需数小时甚至无法处理。
  - **差分隐私（DP）**：噪声添加粗粒度，精度-安全性权衡不佳。
  - **联邦学习（FL）**：无法保护查询隐私和模型隐私，ML模型会直接泄露给云。
- **根本矛盾**：数据加密后变成伪随机数，无法直接用于ML；若不加密则数据安全受损。因此需要在加密数据库上同时实现安全性和ML能力。

## 2. 方法论：核心思想、关键技术细节
### 2.1 核心思想：索引辅助方法
- **索引辅助**：对每个数据项进行正式加密（如AES/3DES）以保证密文隐私，同时生成一个**安全索引项**，该索引项可用于ML训练和查询。索引项由生成器G生成，并通过对抗训练使其与随机数据不可区分，从而保护索引隐私。
- **整体框架**：MLQENABLER包含四个实体：存储客户端、非存储客户端、云服务器、密钥生成服务器（KGS）。KGS训练一个EncGAN模型，其生成器G作为全局密钥；存储客户端用G生成索引项，再用本地密钥加密原始数据，两者一同上传至云；云用索引项训练ML模型；客户端可通过G加密查询令牌后发送ML查询。

### 2.2 关键技术细节：EncGAN
- **网络结构**：包括生成器G、判别器D、重构器R，如图1所示。
- **损失函数**：
  - **对抗损失**（Adversarial Loss）：使G生成的索引项与随机数据不可区分，公式为  
    \( L_{adv}(G,D) = E_{z\sim Unif(X)}[\log D(z)] + E_{x\sim Unif(I)}[\log(1-D(G(x)))] \)
  - **重构损失**（Reconstruction Loss）：使重构器R能从索引项恢复原始数据，保证索引项保留ML所需信息，公式为  
    \( L_{rec}(G,R) = E_{x\sim Unif(I)}[\|R(G(x))-x\|_1] \)
  - **总损失**：\( L(G,D,R) = L_{adv} + \lambda_{rec} L_{rec} \)，训练过程为 \( G^* = \arg\min_{G,R} \max_D L(G,R,D) \)
- **训练算法**：Algorithm 1，交替更新D和G、R。

### 2.3 查询处理流程
- **数据下载查询**：存储客户端凭本地密钥解密。
- **模型下载查询**：授权客户端可获取训练好的模型M'（注意：模型M'仅是G与最终可用模型串联的一部分，G保留在客户端，因此云无法直接使用模型，保护模型隐私）。
- **ML查询**：客户端用G加密查询数据项得到令牌，发送给云，云用M'推理后返回结果。

### 2.4 安全性分析要点
- **密文隐私**：标准加密保证。
- **索引隐私**：G生成的索引项与随机数据不可区分，对抗损失保证。
- **令牌隐私**：与索引隐私相同机制。
- **模型隐私**：云只有M'，而最终可用模型为 \( M = G \circ M' \)，G在客户端，云无法正确使用模型。
