## 摘要
糖尿病护理中使用的连续血糖监测仪 (CGM) 收集了丰富的个人健康数据，可用于改善日常自我管理。然而，目前的患者平台仅提供静态摘要，不支持探究性的用户查询。大语言模型 (LLM) 能够实现对连续血糖数据的自由形式查询，但在敏感健康记录上部署 LLM 会引发隐私和准确性方面的担忧。在本文中，我们提出了 CGM-Agent，这是一个用于个人血糖数据问答的隐私保护框架。在我们的设计中，LLM 仅作为选择分析函数的推理引擎。所有计算均在本地完成，个人健康数据绝不会离开用户设备。为了进行评估，我们构建了一个包含 4,180 个问题的基准测试，结合了参数化问题模板、真实用户查询以及通过确定性程序执行得出的地面真值。通过评估 6 个领先的 LLM，我们发现顶级模型在合成查询上达到了 94% 的数值准确率，在模糊的真实世界查询上达到了 88%。错误主要源于意图和时间歧义，而非计算失败。此外，轻量级模型在我们的智能体设计中也取得了具有竞争力的表现，这表明了低成本部署的可能性。我们开源了代码和基准测试，以支持未来对可信健康智能体的研究。

## Abstract
Continuous glucose monitors (CGMs) used in diabetes care collect rich personal health data that could improve day-to-day self-management. However, current patient platforms only offer static summaries which do not support inquisitive user queries. Large language models (LLMs) could enable free-form inquiries about continuous glucose data, but deploying them over sensitive health records raises privacy and accuracy concerns. In this paper, we present CGM-Agent, a privacy-preserving framework for question answering over personal glucose data. In our design, the LLM serves purely as a reasoning engine that selects analytical functions. All computation occurs locally, and personal health data never leaves the user's device. For evaluation, we construct a benchmark of 4,180 questions combining parameterized question templates with real user queries and ground truth derived from deterministic program execution. Evaluating 6 leading LLMs, we find that top models achieve 94\% value accuracy on synthetic queries and 88\% on ambiguous real-world queries. Errors stem primarily from intent and temporal ambiguity rather than computational failures. Additionally, lightweight models achieve competitive performance in our agent design, suggesting opportunities for low-cost deployment. We release our code and benchmark to support future work on trustworthy health agents.

*   **核心挑战**：
    1.  **隐私风险**：将敏感的健康数据上传至云端大语言模型（LLM）存在严重的数据泄露风险。

### 2. 方法论
*   **核心思想**：**推理与计算解耦（Decoupling Reasoning from Computation）**。LLM 仅作为“推理引擎”来理解意图和选择工具，而实际的数值计算在本地执行。
*   **三层架构流程**：
    1.  **输入处理器（Input Processor）**：作为网关，负责验证问题是否可回答（如：是否超出了 CGM 数据范围），并将模糊的时间表述（如“昨天下午”）解析为精确的时间戳。
    2.  **分析智能体（Analytical Agent）**：将标准化查询分解为多个原子任务，并调用本地 Python 工具包。该工具包包含临床标准指标（TIR、GMI、平均血糖等）和用户驱动的分析（趋势可视化、波动检测）。
    3.  **响应生成器（Response Generator）**：接收本地计算出的**聚合统计结果**（而非原始数据），结合临床背景合成自然语言回答。
*   **隐私保护机制**：原始 CGM 时间序列数据始终保留在本地沙盒中，LLM 仅接触到脱敏后的函数调用指令和高度聚合的指标，从而满足隐私合规要求。

### 6. 主要结论与发现
*   **高性能**：顶级模型在合成查询上达到了 **94%** 的数值准确率，在复杂的真实世界查询中达到了 **88%**。
*   **轻量化潜力**：GPT-5-Mini 和 Gemini 3 Flash 等轻量级模型在工具调用任务中表现出与旗舰模型相当的竞争力，预示着低成本本地部署的可行性。
*   **错误根源**：主要的失败模式是**意图歧义**（如用户对“清晨”的定义与系统不一致）和**时间解析错误**，而非计算逻辑错误。

### 7. 优点
*   **架构优雅**：解耦推理与计算，从根本上解决了 LLM 在医疗领域计算不准和隐私泄露的两大痛点。
*   **临床一致性**：工具包严格遵循 ADA（美国糖尿病协会）等权威机构的临床指南。
*   **基准贡献**：开源了首个结合真实用户访谈与确定性真值的 CGM 问答基准数据集。

### 8. 不足与局限
*   **响应延迟**：目前的流水线涉及多次 LLM 调用，导致响应速度较慢，难以满足实时交互需求。
*   **模态覆盖有限**：虽然支持扩展，但核心实验仍集中在 CGM 数据，尚未全面整合心率、睡眠、连续酮体监测等更多维度的传感器数据。
*   **歧义处理上限**：对于极度模糊的用户输入，系统有时会采取“代理假设”而非多次追问，可能导致结果与用户真实意图存在偏差。
